# IR-3 — Auth-method-per-app

## Purpose

Model authentication as a small set of `AuthMethod` strategies (OAuth2, API key, token, connection string, ...) attached to apps, so "Slack via OAuth or Slack via bot token" is one app with two methods and a user choice, instead of two unrelated credential types.

## BubbleLab today (the problem)

Credential kinds are hand-maintained enum members: `CredentialType` in `packages/bubble-shared-schemas/src/types.ts` (63 members on current main) fanned out through `BUBBLE_CREDENTIAL_OPTIONS` (`credential-schema.ts:2907`, 90 bubble entries). One real-world app can appear as two overlapping systems: `SLACK_CRED` (OAuth) and `SLACK_API` (bot token) sit side by side (`types.ts:21-22`) and both attach to the `slack` bubble, leaving users to guess which to configure. Each bubble also hand-implements `protected abstract chooseCredential()` (`packages/bubble-core/src/types/service-bubble-class.ts:40`) to pick among injected credentials. Note: the improvement register's figure of "141 hand-maintained credential types" does not match the current tree; see Evidence below.

## The delta

One `AuthMethod` strategy per kind, behind the single resolver seam of IR-4. An app declares which methods it supports; a user picks one; the strategy handles acquisition, storage shape, and header/param placement. `SLACK_CRED` vs `SLACK_API` collapses into "Slack: OAuth or token, your choice". On the fork this is the `improve/auth-methods-from-docs` branch (Wave 2): schema + strategies + per-app inference from documentation.

## Why the improvement is needed

Usability and maintenance. Duplicated credential systems for one app confuse users at the exact moment of highest drop-off (connecting an account). On the author side, every new app grows the central enum and maps (the 12-location checklist, IR-1), and every bubble re-implements credential choice by hand. Strategies make the 9 auth kinds a closed, tested set instead of 63-and-growing enum members.

## Reference build vs BubbleLab port

In `integration_stitcher` the strategy model was native: apps declared methods from the start and the resolver seam (IR-4) was the only path to a secret, so adding a kind meant one strategy class. But only 3 of the 9 auth kinds were ever built there; the other 6 are unwritten work, not a port.

In BubbleLab the port must coexist with the enum: 63 `CredentialType` members, the `BUBBLE_CREDENTIAL_OPTIONS` map, the OAuth service (`apps/bubblelab-api/src/services/oauth-service.ts`), and each bubble's `chooseCredential()` all assume the old vocabulary. BubbleLab's design fights it in the per-bubble `chooseCredential()` implementations (github.ts re-calls it inside every operation, e.g. :838, :898, :959, :1021) and in the studio UI keyed to credential types. The pragmatic port maps strategies onto existing enum members rather than deleting them.

## Verdict

PORT the seam and the 3 proven kinds; the remaining 6 kinds are new work to be built and tested, not ported.
