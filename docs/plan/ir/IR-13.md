# IR-13 — Web/DOM contract

## Purpose

Treat a captured DOM interaction (selectors, surrounding structure) as a pseudo-API contract for browser-based integrations: record the exact elements an agent interacted with once, then replay deterministically against that contract, and flag any page that no longer satisfies it as drift instead of silently misreading it.

## BubbleLab today (the problem)

Browser automation exists but with no contract concept. The BrowserBase service bubble (`packages/bubble-core/src/bubbles/service-bubble/browserbase/browserbase.ts`) manages remote browser sessions (session/context ids, CDP cookies), and task-shaped browser tools sit above it (`packages/bubble-core/src/bubbles/tool-bubble/browser-tools/` — linkedin-connection, linkedin-accept-invitations; `amazon-shopping-tool`). What the automation clicked, and whether the page still looks like what it expects, is never captured or checked; a page redesign silently breaks or, worse, silently mis-clicks.

## The delta

A DOM contract captured at authoring time (by an agent recorder, see IR-14's TinyFish-as-recorder idea) and validated at replay time: the replayer checks the page against the recorded selectors/structure before acting, and a mismatch raises drift, feeding the same detect-and-heal loop as API contracts (IR-11/12). The recommended split keeps the LLM out of the run path: a deterministic replayer (`agent-browser`, Apache-2.0, accessibility-tree snapshots) executes against the contract; the paid agent runs only at record time and as a healer when drift fires.

## Why the improvement is needed

Cost and silent-failure risk. Agent-driven browsing on every cron fire is ruinous (TinyFish bills ~$0.015 per agent step) and non-deterministic; deterministic replay is near-free but brittle without a contract, because a changed page yields wrong clicks with no error. The contract converts "the page changed" from an invisible hazard into a detected event, which is the same invariant the API-side contracts hold.

## Reference build vs BubbleLab port

In `integration_stitcher` the DOM-contract logic was built and its drift detection validated: a replay that no longer satisfies the captured contract is flagged, not silently misread. But the validation ran against captured/mock DOM only, the sole browser driver ever built was a mock, so no real browser has ever exercised the contract.

In BubbleLab nothing fights it structurally (the BrowserBase bubble is one integration among many), but nothing supports it either: there is no recorder, no replayer, and no real driver anywhere in either codebase. The port is blocked behind IR-14's re-analysis; scheduling it before a real driver exists would be building the second story first.

## Evidence / status

PARTIAL / LAB-ONLY. Drift-flagging logic validated in the reference build against captured DOM; never driven by a real browser. Port status: deferred (status map: "comes with browser"), explicitly not scheduled before the API-side wins. Feasibility caveat carried from HANDOFF §7: confirm TinyFish's API returns the interacted-with element/DOM, not just extracted data; if not, the contract must be captured from page snapshots ourselves.

## Verdict

RE-ANALYSE — the contract shape is right and lab-validated, but blocked on a real browser driver (IR-14); do not schedule before the API-side work lands.
