# IR-9/10 — The Tester (probe-to-ground) + auto-run gate

## Purpose

Ground contracts in observed reality instead of documentation fiction, and gate risky execution: reads run for real (recording the true response); writes mock by default; a real write happens only under an explicit per-operation grant. This is the register's highest-value item, the one that breaks BubbleLab's "build never executes" wall.

## BubbleLab today (the problem)

Nothing in the pipeline calls a real API before run time. The code-generating LLM grounds on synthesized fiction: `get-bubble-details-tool.ts` fabricates example values from the Zod schema (`generateExampleValue()` returns the literal `"example string"` for any string and `42` for any number), and `MockDataGenerator` (`packages/bubble-shared-schemas/src/mock-data-generator.ts`) synthesizes mock results the same way. (Attribution note: the HANDOFF pins the `"example string"`/`42` literals to `mock-data-generator.ts:44`; on the current tree those exact literals live in `get-bubble-details-tool.ts`, with MockDataGenerator producing equivalent schema-derived fiction. The substantive claim, that no real response ever grounds generation, is verified.) Contract drift therefore surfaces only when production explodes.

## The delta

The port reframes the reference design under the owner's two-phase model (REPO-MAP §4b), which keeps BubbleLab's "authoring never executes" property instead of breaking it:

- Phase 1 (authoring): mock contracts only, zero credentials, zero calls. No read-scoped or probe credential exists anywhere.
- Phase 2 (explicit TEST run or first real run): the sign-off gate. The server computes the flow's write set from `BubbleParser` call sites resolved against IR-8 metadata; a non-empty unapproved write set returns `409 {pendingApproval}` and dispatches nothing. Approval carries `approvedWriteCallSites` (exact call-site match, no wildcards). Reads run for real and record true responses; the base class mocks unapproved writes even if a client lies ("base class is law").

The mechanical half is already merged (PR #3, the test-mode switch): interception ABOVE `performAction` in `BaseBubble.action()` (`base-bubble-class.ts`, gate before the `performAction` call at :317), so no client is constructed and no credential read, auth-agnostic across SDK-client bubbles (`resend.ts:363`) and raw-fetch bubbles (`github.ts`). `POST /:id/test` exists. The grounding half (recording real reads, the sign-off UX, docs-lie reclassification) is branch `improve/run-grounding-and-signoff`, pushed and awaiting merge.

## Why the improvement is needed

Correctness and safety at once. Generation grounded on `"example string"` produces flows that type-check against fiction and break on the first real response shape. And without a write gate, the only way to test a flow is to run it against production systems, sending real emails and mutating real records. The reference evidence quantifies both: 4 operations whose docs said "read" were caught mutating state (docs lie; only execution catches them), and default mode produced 0 production mutations across all 48 scenario cells.

## Reference build vs BubbleLab port

In `integration_stitcher` the Tester sat between validate and run as a build-time probe with its own read-scoped credential mode, natural in a greenfield where the pipeline was designed to execute.

In BubbleLab the same outcome is grafted through seams that already existed: the sealed `action()` lifecycle (which already short-circuited on `previousResult`) gained the mock gate, and `executionMeta` (already threaded into every bubble constructor by the injector) carries `testMode`, so zero new source rewriting was needed (risk #1). The reframe to two-phase + sign-off is a deliberate deviation from the reference design: it kills the need for read-scoped credentials and preserves the no-execution authoring property BubbleLab's architecture assumes. What still fights: the codegen prompts must be taught test-mode semantics across all codenamed generation paths (`boba/milktea/pearl/coffee/rice`, risk #6), not yet done.
