# IR-4 — Credential resolver seam

## Purpose

All credential resolution flows through one seam: a single interface that takes "which credential does this call site need" and returns a usable secret, so the provider behind it (DB, vault, test double) can be swapped with zero changes upstream.

## BubbleLab today (the problem)

Resolution is spread across three mechanisms with no single seam (verified in `docs/plan/HANDOFF.md` §4): discovery in `BubbleInjector.findCredentials()` (`packages/bubble-runtime/src/injection/`), injection by source-text rewriting in `BubbleInjector.injectCredentials()` (`BubbleInjector.ts:447`), and secret resolution at execution via `apps/bubblelab-api/src/services/credential-helper.ts` → `oauth-service.ts` `getValidToken()` (:347). Below that, two incompatible consumption patterns coexist: SDK-client bubbles construct clients inside `performAction` (`resend.ts:363`, `:379` — `new Resend(apiKey)`), and raw-fetch bubbles have no client at all, re-calling `chooseCredential()` inside every operation (`github.ts:838`, `:898`, `:959`, `:1021`, ...). "Swap the client for a mock" therefore has no single hook; any per-client interception would be per-bubble, per-auth-type work.

## The delta

One resolver interface at the point where the runtime turns a credential id into a secret. Auth strategies (IR-3) plug in behind it; refresh policy (IR-5) lives inside it; test doubles replace it wholesale. Upstream code (injector, bubbles) never learns where secrets come from.

## Why the improvement is needed

Maintenance and testability. Without a seam, every credential-behavior change (refresh policy, locking, vault migration, test mocking) is a hunt across `credential-helper.ts`, `oauth-service.ts`, and per-bubble code. The reference build's provider-swap test is the concrete payoff: swapping the provider required zero upstream change. Note the fork already learned the companion lesson: the test-mode switch (merged PR #3) deliberately intercepts ABOVE `performAction` rather than at the client, precisely because there is no uniform client to hook; the resolver seam solves the same fragmentation one layer down, for runs that do execute.

## Reference build vs BubbleLab port

In `integration_stitcher` the seam was the founding abstraction of the credentials package: nothing else ever touched a secret, so the swap test was trivial to write and passed by construction.

In BubbleLab the seam must be retrofitted between `credential-helper.ts`/`oauth-service.ts` and their callers. BubbleLab's design half-helps: resolution is already funneled through `getValidToken()` for OAuth, so the seam has a natural home; it fights back in the raw-fetch bubbles, where `chooseCredential()` reads from injected params rather than asking a resolver, and in the injector's source rewriting (IR-15), which moves credential ids through text instead of through a context object.

## Evidence / status

CONFIRMED: provider swap with zero upstream change in the reference build. Port status: building (Wave 2, together with IR-3 on branch `improve/auth-methods-from-docs`).

## Verdict

PORT — the seam is proven and gives IR-3 and IR-5 their home; effort rides with the IR-3 branch.
