# IR-2 — Author-ease / user-control levers

## Purpose

Give integration authors and end users explicit control surfaces: 7 author levers (e.g. per-operation scope declaration, side-effect hints, contract strictness) and 7 user levers (e.g. per-operation scope picker, a read/write dial, credential granularity choices), so safety and capability are chosen, not inherited by accident.

## BubbleLab today (the problem)

Credential entry is raw: users paste API keys and multi-part credentials into forms driven by `BUBBLE_CREDENTIAL_OPTIONS` (`packages/bubble-shared-schemas/src/credential-schema.ts:2907`) and the `CredentialType` enum (`types.ts`). There is no per-operation scope picker (scopes are requested at OAuth time and stored, `apps/bubblelab-api/src/services/oauth-service.ts:524`, but the user never selects per operation) and no read/write dial (before IR-8 landed on the fork, bubbles carried no read/write concept at all).

## The delta

The proposed lever set: 7 author-side and 7 user-side controls layered over the auth and metadata systems (IR-3/4 for auth kinds, IR-8 for side-effect hints, IR-6/7 for scopes). Each lever is a declared, inspectable setting rather than an emergent property of whatever credential was pasted.

## Why the improvement is needed

Usability and least-privilege. A user connecting Slack today grants whatever the OAuth config requests, with no way to say "reads only" or "these three operations". Authors have no vocabulary to declare what a well-scoped connection looks like. The counter-argument, and the reason this IR is unproven: no user need was ever validated for the specific 14-lever shape. Some of its substance has since been delivered by other IRs (the read/write dial exists as the test-mode gate; scope declaration arrives with IR-6/7), which weakens the case for building the rest as designed.

## Reference build vs BubbleLab port

In `integration_stitcher` the levers were designed greenfield as part of the auth and contract model, but the claim was never exercised: no test was written against any lever, so "greenfield-native" here means "sketched in a codebase that made it easy", not proven.

In BubbleLab a port would touch the studio credential UI, the OAuth request path, and per-operation metadata at once, cutting across `apps/bubble-studio` state and `apps/bubblelab-api` OAuth routes. BubbleLab's design fights it wherever a lever needs per-operation granularity, because credentials today attach per bubble (via `BUBBLE_CREDENTIAL_OPTIONS`), not per operation. Building 14 levers on an unvalidated need against that grain is the wrong order.

## Evidence / status

RE-ANALYSE. No test was ever written in the reference build; the claim was never validated. The register and the status map both exclude it from porting. Partial overlap with shipped work: the fork's test-mode gate (merged PR #3) and IR-8 metadata (merged PR #2) deliver a server-enforced read/write control without any of the lever UI.

## Verdict

RE-ANALYSE — re-derive from a real user need before building; do not port on faith.
