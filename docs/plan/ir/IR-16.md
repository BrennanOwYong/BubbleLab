# IR-16 — Sandboxed execution

## Purpose

LLM-generated flow code executes inside a worker/isolate with an allowlist of capabilities, so the code can reach only what it is granted, instead of running with the API server's full Node privileges behind a bypassable denylist.

## BubbleLab today (the problem)

`BubbleRunner` writes the generated code to a temp `.ts` file and dynamically imports it into the API server's own process: temp file created at `BubbleRunner.ts:442`, `sanitizeScript()` applied at :446, `import(moduleUrl)` at :466, unlink at :695. The guard is a denylist with two layers, both pattern-based: at runtime, `sanitize-script.ts:10` regex-replaces `process.env` accesses (including `process['env']` variants, with string/comment protection) with a throw; at validation, an AST lint rule (`validation/lint-rules.ts:633`) rejects `process.env` property accesses. Neither restricts anything else: the imported module runs in-process with full access to the filesystem, network, `child_process`, globals, and every other env-reaching trick a denylist misses (`Reflect.get(globalThis.process, 'env')`, dynamic property names, a dependency that reads env for you). A denylist on a live code-execution surface is a bypassable guard on the crown jewels: the process that holds `CREDENTIAL_ENCRYPTION_KEY` and the DB connection.

## The delta

Execute generated code in a worker/isolate with an allowlist: the flow receives exactly the capabilities it is granted (the bubble runtime, its resolved credentials via context) and can reach nothing else. No temp file, no dynamic import of generated code into the server process.

## Why the improvement is needed

Security, and the register is explicit that this is a genuine security fix, not a nicety. The threat model is not hypothetical: the code being executed is written by an LLM from user prompts, so prompt injection is an arbitrary-code-execution path into the API server. Denylists fail open (every bypass is a full compromise); allowlists fail closed. The in-process import also has operational costs the REPO-MAP notes: `import()` caches by URL and temp files on WSL/`/mnt/c` are slow and Windows-locked, producing flaky unlinks.

## Reference build vs BubbleLab port

In `integration_stitcher` isolation was the execution model from the start, so the evidence test (env and disk unreachable except via allowlist) described the architecture rather than patching it.

In BubbleLab this is surgery on the runner's core: the temp-file-import mechanism is how every flow runs, and the imported module links directly against `bubble-core` classes in the same process, which is exactly what the merged test-mode work relies on (interception lives in base classes the imported module links against, REPO-MAP risk #2). An isolate boundary changes how bubbles, context, and results cross between runner and flow, so it interacts with IR-15 (context injection) and is sequenced with it in the deferred-surgery pile.

## Evidence / status

CONFIRMED in the reference build: `process.env` and disk unreachable except via allowlist. Port status: not scheduled (status map: SURGERY, deferred). Current-state correction to the register's phrasing: "regex-blocks `process.env`" is accurate for the runtime sanitizer, and there is additionally an AST-based lint rule at validation; both are denylists, so the register's conclusion stands.

## Verdict

SURGERY — effort HIGH; a genuine security fix; do deliberately with IR-15/17, after the additive wins.
