# IR-14 — Browser observe-and-intervene

## Purpose

Browser automation runs as a supervised session a human can watch and take over mid-run, with the whole session captured as a replayable trace, instead of a black box that either succeeds or fails invisibly.

## BubbleLab today (the problem)

Browser automation is hidden inside tools. The BrowserBase service bubble (`packages/bubble-core/src/bubbles/service-bubble/browserbase/browserbase.ts`) provisions remote sessions, and tool bubbles drive them for fixed tasks (`tool-bubble/browser-tools/linkedin-connection/`, `linkedin-accept-invitations/`, `amazon-shopping-tool/`). No supervision surface, no human takeover, no trace capture, no DOM contract (IR-13). When a browser task goes wrong, the user learns from the outcome, not during the run.

## The delta

A supervised session model: the user observes the browser live, can intervene (take over the mouse, complete a CAPTCHA or 2FA step, correct a mis-click), and the session, including the human's interventions, is captured as a replayable trace. Combined with IR-13, the current framing is "record once, replay forever": an agent (TinyFish) performs the action once at authoring time and reports the exact HTML it interacted with; that capture becomes the DOM contract; a deterministic replayer (`agent-browser`) runs it thereafter at ~zero cost, and the agent returns only as a healer when drift is detected.

## Why the improvement is needed

Trust and coverage. Many business apps have no API, so browser automation is the only wrapper, but unsupervised agent browsing on real accounts (LinkedIn, checkout flows) is a liability users reasonably refuse. Observe-and-intervene bounds the blast radius (a human is watching the run that matters) and the trace makes agent behavior auditable and repeatable. The recorder/replayer split also fixes the economics: TinyFish's ~$0.015 per agent step is ruinous per cron fire, acceptable once at record time.

## Reference build vs BubbleLab port

In `integration_stitcher` the observe-and-intervene machinery was built, but against a mock: the only browser driver that ever existed is a mock driver. No real browser was ever driven, so the entire IR is design work with untested integration assumptions, including whether takeover mid-agent-run is even cleanly possible with the candidate drivers.

In BubbleLab nothing structural blocks it (it would be a new execution surface beside the BrowserBase bubble, not surgery on the runtime), but nothing validates it either. The port-relevant unknowns are external: whether TinyFish's API returns the interacted-with element/DOM rather than just extracted data (HANDOFF §7's explicit must-verify), and whether `agent-browser`'s accessibility-tree snapshots suffice as the replay contract.
