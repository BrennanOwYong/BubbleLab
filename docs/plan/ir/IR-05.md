# IR-5 — Refresh-on-expiry + single-flight lock

## Purpose

OAuth tokens refresh only when expired (with a safety buffer), and concurrent resolutions of the same expired token produce exactly one refresh call, ending token churn and refresh races.

## BubbleLab today (the problem)

`getValidToken()` (`apps/bubblelab-api/src/services/oauth-service.ts:347`) refreshes on every single resolution regardless of expiry. The code says so itself: the doc comment reads "Get a valid access token, always refreshing to ensure freshest credentials", and the body comment "Always refresh if we have a refresh token" precedes an unconditional `refreshToken()` call whenever a refresh token exists. Token rows already carry `oauthExpiresAt` (checked inside `refreshToken()` at :385 only as a presence guard, never as an expiry test). There is no lock of any kind: parallel resolutions, or parallel API instances, race `refreshToken()` and can rotate each other's refresh tokens with providers that single-use them.

## The delta

Refresh only when `now > expiresAt - buffer` (300 s per the REPO-MAP plan); wrap the refresh in a single-flight guard (module-level `Map<credentialId, Promise<string>>` so concurrent callers await the same promise), with a DB advisory lock for multi-instance deployments. Keep the existing fall-back-to-stored-token on refresh failure. The other resolution path, `credential-helper.ts`, routes through `getValidToken` and inherits the fix.

## Why the improvement is needed

Cost, latency, and correctness. Every flow run today pays a provider round-trip per OAuth credential even when the token has hours of life left, adding latency to every execution and load to provider rate limits. The race is worse than waste: providers that rotate refresh tokens on use (Google does) can invalidate the token a concurrent racer just stored, producing intermittent "user must re-auth" failures with no code bug to find.

## Reference build vs BubbleLab port

In `integration_stitcher` expiry-aware refresh with a single-flight lock was the only token path ever written, sitting inside the resolver seam (IR-4), so the evidence tests (0 refreshes on a valid token; 1 refresh for 10 concurrent resolutions) were written directly against the seam.

In BubbleLab the change is surgical and local to `oauth-service.ts`, the friendliest port in the register: the function exists, the expiry timestamp is already persisted, and callers are unaffected. The only design friction is multi-instance deployment (module-level locks do not cross processes), which the port answers with a DB advisory lock on the credential row.

## Evidence / status

CONFIRMED in the reference build: valid token → 0 refreshes; 10 concurrent resolutions of an expired token → exactly 1 provider call. Port status: building (Wave 1, branch `improve/ir5-refresh-on-expiry`, not yet pushed as of 2026-07-15).

## Verdict

PORT — effort LOW, isolated, zero architectural risk; the register names it "do this first".
