# IR-6/7 — Proactive scope audit + honest fallback

## Purpose

A flow fails at build/validation time, naming the exact missing scope, when a required OAuth scope was never granted, instead of failing mid-run when the provider rejects the call. Where a provider exposes no scope metadata, the system says so explicitly rather than pretending to have checked.

## BubbleLab today (the problem)

Scope enforcement is reactive only. Granted scopes are persisted on the credential row (`apps/bubblelab-api/src/services/oauth-service.ts` — `scopes` fields at :29/:47/:143, stored as `oauthScopes` at :524) but nothing audits them: flow validation (`apps/bubblelab-api/src/services/validation.ts`, `credential-validator.ts`) checks credential presence, not capability. A missing `chat:write` surfaces as a provider 403 in the middle of a production run, attributed to nothing.

## The delta

At flow validation, union each parsed call site's `requiredScopes` (declared in IR-8's per-operation metadata; `BubbleParser` already yields bubble + operation per call site), diff against the granted set on the credential row, and fail the build naming the missing scope. Providers without scope metadata degrade honestly: an explicit "scopes unknown for this provider, will surface on first run" message, never a silent pass presented as a check.

## Why the improvement is needed

Usability and trust. A scope failure discovered mid-run can leave a flow half-executed (three writes done, fourth rejected), and the provider's 403 text rarely names the scope. Catching it before any call runs converts a production incident into a one-line fix at author time ("grant chat:write and reconnect"). The honest-fallback half exists because a checker that stays silent when it cannot check trains users to trust a guarantee it is not giving.

## Reference build vs BubbleLab port

In `integration_stitcher` operations declared `requiredScopes` from birth inside the contract, so the audit was a set-difference over data that always existed, and the honest fallback was a contract field ("scope metadata absent") rather than a special case.

In BubbleLab the audit slots into an existing validation pass, but the data must be backfilled: IR-8's schema declares `requiredScopes` as optional and the merged 59-operation backfill did not populate it yet (see `docs/plan/deltas/ir8-side-effect-metadata.md`, "declared (optional) ... but not yet populated"). BubbleLab's design also fights the mapping itself: credential types, not operations, are the granted-scope unit, and per-provider scope vocabularies differ (REPO-MAP risk #4), so expect per-provider special cases.

## Evidence / status

CONFIRMED in the reference build: an ungranted scope fails the audit naming `chat:write`; metadata-less providers degrade with an explicit first-run message. Port status: in progress (Wave 2, branch `improve/ir6-scope-audit`, pushed; core module + Google-bubble scopes done, API wiring in progress as of 2026-07-15). Blocked on `requiredScopes` population for full coverage.

## Verdict

PORT — effort LOW–MEDIUM once operations declare scopes; the audit logic itself is cheap.
