# IR-8 — Doc-grounded side-effect classifier

## Purpose

Every operation knows whether it reads or writes, derived from the vendor's documentation (never the HTTP method), carrying `{confidence, source, citation}` so the classification is auditable back to the doc sentence that produced it. This is the prerequisite for the test-mode gate, the Tester (IR-9/10), the scope audit (IR-6/7), and the Contract KB (IR-11/12).

## BubbleLab today (the problem)

Upstream BubbleLab has no read/write concept at all: nothing declares whether an operation mutates, so nothing can reason about safety before running. This is now historical for the fork: IR-8 merged to fork main as PR #2 (commits `b9676d4`, `c778b94`, `cbca070`). Upstream remains without it.

## The delta

Per-operation `sideEffect: 'read' | 'write' | 'read_with_side_effects'` (plus `destructive`, `idempotent`) in bubble static metadata. Landed on the fork as: `OperationSideEffectMetadataSchema` (`packages/bubble-shared-schemas/src/operation-metadata-schema.ts`, citation `min(1)` so no classification exists without its source); the classifier (`packages/bubble-core/src/utils/side-effect-classifier.ts`) with evidence hierarchy `mcp > openapi > prose > manual` and a binding rule (write iff the docs say a new record is created; update/delete/mark → `read_with_side_effects` + `destructive`; no signal → `ClassificationError`, never a guess); `ServiceBubble.static operationMetadata` (`service-bubble-class.ts`) resolved at runtime by `BaseBubble.get sideEffect()` with fail-safe default `'write'`; catalogue surfacing through `get-bubble-details-tool` and `bubbles.json`; and a 59-operation, 6-bubble backfill in colocated `<name>.metadata.ts` files.

## Why the improvement is needed

Security and correctness of everything downstream. Without a machine-readable mutation hint, a test run cannot know which operations are safe to execute, a scope audit cannot rank risk, and a codegen LLM cannot be told which calls have consequences. The method is never the signal: the evidence base caught a POST that only reads and a GET that creates, and 4 operations whose docs said "read" were caught mutating state in the reference build's live probes. Deriving from docs with citations makes every classification contestable and correctable, which the `observed`-source runtime correction channel (REPO-MAP §4a step 5) depends on.

## Reference build vs BubbleLab port

In `integration_stitcher` the classifier lived in the contracts package and classified operations at generation time from the same docs that produced the contract, one pipeline, one pass, 19 tests.

In BubbleLab the port had to answer a question the greenfield never faced: ~60 existing hand-written bubbles with no metadata. The answers became design features: fail-safe `'write'` for anything unclassified (the ~55 un-backfilled bubbles are safe by construction under the test-mode gate), colocated metadata files to avoid a 13th central registry location (REPO-MAP risk #3), and a rerunnable backfill script with a CI guard so new operations cannot ship unclassified. BubbleLab's discriminated-union-on-`operation` params convention helped rather than fought: it gave the classifier its per-operation key for free.

## Evidence / status

CONFIRMED and MERGED (fork main, PR #2). Reference evidence: 19 tests, POST-that-reads → `read`, GET-that-creates → `write`. Port evidence: 21 new tests pass (`side-effect-classifier.test.ts` 14/14, `operation-metadata.test.ts` 7/7) including the same method-is-not-the-signal acceptance cases; backfill covers resend (3 ops), gmail (17), google-calendar (6), google-drive (12), github (10), airtable (11), 59 operations, zero unverified, every entry cited to a vendor doc URL. Gate: build PASS, typecheck PASS; the 12 test failures in the full run are pre-existing WSL `/mnt/c` timeouts reproduced on unmodified main (see the delta record).
