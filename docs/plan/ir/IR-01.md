# IR-1 — Declarative AppSpec (kill the 12-location checklist)

## Purpose

Adding an integration becomes dropping one declarative spec file that the system auto-discovers. Zero edits to central registries, enums, maps, or exports. The spec declares what the app is (name, auth, operations, schemas); the factory derives everything else.

## BubbleLab today (the problem)

Adding one integration requires editing 12 separate locations, documented by BubbleLab themselves in `packages/bubble-core/CREATE_BUBBLE_README.md:1050` ("NEW BUBBLE REGISTRATION CHECKLIST ... you must update these **12 locations**"): the `CredentialType` enum (`packages/bubble-shared-schemas/src/types.ts`), the credential configuration map (`bubble-definition-schema.ts`), `BUBBLE_CREDENTIAL_OPTIONS` (`credential-schema.ts:2907`), env mapping, factory registration (`packages/bubble-core/src/bubble-factory.ts`, `register()` at :87, defaults from :481), package exports, studio UI config, logo assets, and more. Integrations are monolithic hand-written files (`slack/slack.ts` is 124 KB). This checklist is BubbleLab's headcount ceiling: every new app costs an engineer walking 12 files.

## The delta

A declarative AppSpec plus auto-discovery: one file per integration, discovered at factory init, no central edits. It coexists with hand-written bubbles, both register into the same `BubbleFactory`, so adoption is incremental, integration by integration. On the fork this lands through the contract-first generation work (branch `improve/contract-first-generation`), which generates the spec from the app's documentation rather than asking a human to write it.

## Why the improvement is needed

Maintenance ceiling and error surface. Twelve hand-edited locations means twelve chances to forget one, and merge contention on shared registry files whenever two integrations land at once (the IR-8 backfill already had to design around this, see `docs/plan/deltas/ir8-side-effect-metadata.md`, "no 13th central registry location"). Scaling to "wrap every business app" is impossible while each app costs a checklist walk; a spec-file-per-app makes integration count a function of doc quality, not headcount.

## Reference build vs BubbleLab port

In `integration_stitcher` this was greenfield-native: the package layout was designed around spec files from day one, so discovery was the only registration mechanism and nothing competed with it.

In BubbleLab it is grafted onto brownfield: the 12 locations exist and keep working, so the AppSpec path must register into the same factory alongside them without becoming location 13 itself. BubbleLab's design fights it in two places: the central `CredentialType` enum (new apps need credential kinds, which today means an enum edit, so full zero-central-edit requires IR-3/IR-4's auth-method work) and the studio UI configs that key off central maps. The colocated-file pattern proven by the IR-8 backfill (`<name>.metadata.ts` next to each bubble) is the template for keeping the property.

## Evidence / status

CONFIRMED in the reference build: 5 integrations discovered with zero central edits and zero network calls during generation. Port status: scheduled (Wave 3, branch `improve/contract-first-generation`, building as of 2026-07-15). Nothing landed on the fork's main yet. Note the related contract-generation caveat: the reference path requires OpenAPI 3.x input; the agent-reads-prose-docs front end is unbuilt.

## Verdict

SURGERY, but incrementally adoptable (coexists with hand-written bubbles; adopt one integration at a time). Effort HIGH.
