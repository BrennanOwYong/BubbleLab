# IR-15 — Runtime-context injection, not source rewriting

## Purpose

Credentials, logging, and per-run state reach bubbles through a runtime context object handed to the flow, so the generated source is executed exactly as written. The source-text rewriter is deleted.

## BubbleLab today (the problem)

`BubbleInjector` (`packages/bubble-runtime/src/injection/BubbleInjector.ts`) rewrites the source text of every bubble instantiation: `injectCredentials()` (:447) and `injectCredentialsIntoBubble()` (:678) splice `credentials: { SLACK_CRED: 2 }` (a DB id, never a secret) into each `new XBubble({...})` expression, with line-array `splice` operations (:992, :1044) and line-shift tracking, then params are re-read back out of the modified source. `LoggerInjector.ts` does the same for logging. The register calls this their most fragile code, and the fork's own builds treat it that way: REPO-MAP risk #1 forbids threading any new per-run state through the rewriter, and the merged test-mode switch deliberately traveled through `executionMeta` (which the injected constructor context already threads into every bubble via `parameter-formatter.ts`) precisely to avoid adding to it.

Scoping note (the AST retraction, HANDOFF §2): the parsing side is not the problem. `BubbleParser` uses a real AST (`@typescript-eslint/parser`) with per-call-site identity; the fragility is the write path, mutating source text and tracking line shifts, not the read path.

## The delta

Pass credentials and logging through a runtime context object: the runner resolves the credential mapping per call site (identity already exists) and hands it to the flow in context; bubble constructors read from context instead of from spliced params. The rewriter is then deleted. Step source stays byte-identical before and after a run.

## Why the improvement is needed

Maintenance ceiling and correctness risk. Text splicing with line-shift bookkeeping breaks in creative ways as the generated code's shapes evolve (multi-line constructor args, nested instantiation, formatting changes), and every new per-run concern either bloats the rewriter or has to route around it. The workaround channel already exists (`executionMeta` threading proves the context path works); the rewriter persists only for credentials and logging. Byte-identical source also makes runs auditable: what was validated is exactly what executed.

## Reference build vs BubbleLab port

In `integration_stitcher` there was never a rewriter to delete: context injection was the only mechanism, so the evidence (step source byte-identical before/after a run; no `eval`, no `new Function`) held by construction.

In BubbleLab this is deep surgery: every bubble construction site and the whole discover/inject/choose credential flow (HANDOFF §4) assumes spliced params, and `chooseCredential()` implementations read from `this.params`. The port must move ~60 bubbles' credential consumption from params to context, or shim params-from-context, while the injector's other job (parameter re-reading for the flow visualizer) is disentangled. BubbleLab's design fights it everywhere the injector's output is treated as the source of truth.

## Evidence / status

CONFIRMED in the reference build: step source byte-identical before/after a run; no `eval`, no `new Function`. Port status: not scheduled (status map: SURGERY, deferred, "do last, deliberately"). Interim policy already enforced on the fork: no new state goes through the rewriter.

## Verdict

SURGERY — effort HIGH; touches every bubble construction site; do last, deliberately, after the additive wins land.
