# ADD-ANY-APP — automatic onboarding of an app with no existing integration

Feature design. Input: an app name plus a docs URL (or spec URL, or MCP server URL).
Output: a compliant, registered, doc-grounded BubbleLab integration, with no hand-coding.
Companions: `BACKFILL-METHOD.md` (the side-effect method this pipeline embeds),
`HANDOFF.md` §5/§8, `IMPROVEMENT-REGISTER.md` (this is the "CG, contract-first integration
generation" item, upgraded with the prose front-end the register lists as "unbuilt, the true
unlock"), `REPO-MAP.md` §3/§4.

Two audiences, one pipeline:
- **Ops teams**: point-and-click "Add app" in bubble-studio; REST/SaaS apps
  (docs URL in, integration out).
- **Technical teams**: warehouse/SQL-shaped apps (Snowflake, Databricks), CLI-first, carrier
  operations (`execute_statement`), token/JWT credentials with a host/account field.

The two worked examples below are the two hard doc-acquisition cases, researched against the
official sources (References at bottom):
- **Snowflake**: SHIPS official OpenAPI 3.0 specs (`snowflakedb/snowflake-rest-api-specs`,
  40+ files including `sqlapi.yaml`), plus prose docs, plus drivers/SDKs (the Python API is
  itself generated from these specs). Pipeline takes the spec path.
- **Databricks**: ships NO public OpenAPI. `docs.databricks.com/api` is a JavaScript-rendered
  SPA (single-page application) that returns an empty HTML shell to plain fetches (verified by
  probe). Official operation prose, verbs, and paths live in the generated SDKs
  (`databricks-sdk-go` et al., "generated by Databricks SDK Code Generator"). Pipeline takes
  the prose/SDK path.

---

## 1. The compliance target: what "a BubbleLab integration" is

A generated app must be indistinguishable in shape from a hand-written bubble
(`resend.ts`, `github.ts` are the two canonical references):

```
packages/bubble-core/src/bubbles/service-bubble/<app>/
├─ <app>.schema.ts       Zod: params discriminated union on `operation` + result schemas
├─ <app>.ts              class <App>Bubble extends ServiceBubble<Params, Result>
├─ <app>.metadata.ts     BubbleOperationMetadata (doc-grounded side-effect map, cited)
├─ <app>.utils.ts        request builder / helpers (optional)
├─ index.ts              exports
└─ <app>.test.ts         unit tests (schema round-trips, metadata coverage)
```

Class contract (from `types/service-bubble-class.ts` + `CREATE_BUBBLE_README.md`):

| Member | Generated from |
|---|---|
| `static bubbleName / service / alias / shortDescription / longDescription` | App name + doc summaries |
| `static authType: 'oauth' \| 'apikey' \| 'none' \| 'connection-string'` | Auth inference (stage S5) |
| `static schema` (discriminated union on `operation`), `static resultSchema` | Contract derivation (S3), every field `.describe()`d from doc prose |
| `static operationMetadata` | Side-effect classification (S4), imported from `<app>.metadata.ts` |
| `performAction(ctx)` switch over operations | Request mapping (S6): fetch-based, per-op `toRequest` bindings |
| `chooseCredential()` | Credential type mapping (S5); github.ts pattern (raw fetch + per-op credential read) |
| `testCredential()` | The cheapest classified-`read` operation, generated as the probe |

Design choice: generated bubbles are **raw-fetch** (github.ts pattern), never vendor-SDK-client
(resend.ts pattern). Reasons: zero new dependencies per app, one shared request-builder template,
and the REST surface is what the docs/spec describe. For Snowflake/Databricks specifically the
REST surfaces (SQL API, Statement Execution API) replace any need for their drivers.

## 2. Input taxonomy

```
AddAppRequest = {
  appName: string,
  source:
    | { kind: 'openapi', url | file }        // best case
    | { kind: 'mcp', serverUrl }             // annotations + tool schemas for free
    | { kind: 'prose', docsUrl[] }           // most real business apps
    | { kind: 'sdk', repoUrl }               // SPA-doc vendors with generated SDKs (Databricks)
  archetype?: 'operations' | 'carrier',      // carrier = run-SQL/run-code shaped
  auth?: hints (env var names, oauth client source),
  select?: operation allowlist / pick fields  // ops teams: checkbox list in studio
}
```

The pipeline auto-detects what the source really is (S1 discovery), so a bare
`{ appName, docsUrl }` is a valid full input.

## 3. Pipeline stages

```
S1 acquire docs → S2 extract operations → S3 derive contracts (Zod)
→ S4 classify side effects (+cite) → S5 infer auth → S6 generate code
→ S7 validate → S8 register
```

Every stage emits a reviewable artifact; the pipeline is re-runnable per stage (idempotent
regeneration, same file targets).

### S1 — Doc acquisition (the DocPack)
Discovery order, first hit wins per operation set (BACKFILL-METHOD §4):
1. User-supplied OpenAPI URL/file.
2. Vendor spec repo probe (GitHub `<vendor>/*api-specs*`, `/openapi.json`, docs download links).
3. MCP server `tools/list` (schemas + `ToolAnnotations`).
4. Static-HTML prose docs crawl of the supplied docs URL(s).
5. SPA fallback: official generated-SDK docstring mining (Databricks case), else
   headless-browser render of the docs pages.

Output `DocPack`: raw documents + per-document `{url, retrievedAt, sha256}`. The DocPack is
cached in the pipeline workspace (NOT committed) and every later citation points into it, so a
re-run can diff vendor-doc drift.

### S2 — Operation extraction (normalize to OperationDraft)
One shape regardless of source:

```ts
OperationDraft = {
  name: string,                      // snake_case operation id
  http?: { method, pathTemplate },   // absent only for MCP-native tools
  doc: { prose: string, citation: string },
  inputSchema: JsonSchema,           // flattened: path+query+header params + body properties
  outputSchema?: JsonSchema,         // success response, when documented
  requiredScopes: string[],
  securitySchemeName?: string,
}
```

- **OpenAPI path (deterministic, proven).** Port of the reference build's `generateAppSpec`
  front half (`integration_stitcher/packages/contractgen/src/generate.ts`): `composeInput`
  flattens parameters + JSON body into one object schema with wire bindings
  (path/query/header/body/rawBody) and collision checks; `successResponseOf`/`planOutput` pick
  the documented 2xx response; `securityPlanOf` lifts scopes + scheme per operation. Evidence:
  0 network calls, contract-derived schemas validated live responses first-try, 251 ms doc to
  live tool (IMPROVEMENT-REGISTER, CG).
- **MCP path.** `tools/list` already returns name + JSON Schema input + annotations; map 1:1.
- **Prose path (LLM-assisted, gated).** An extraction agent reads DocPack pages and emits
  OperationDrafts with a verbatim quote per claim (operation exists, param name/type/required,
  response field). Same grounding check as BACKFILL-METHOD §6: a quote that is not a substring
  of the DocPack page rejects the draft field. Prose-derived drafts always pass the S7 human
  review gate before code generation.
- **SDK path.** Deterministic docstring mining where the vendor generates SDKs from a spec:
  for Databricks, `service/*/api.go` interface docstrings give the official prose and
  `service/*/impl.go` gives exact verb + path per method (verified below). This is parsing, not
  guessing; treat it as prose-tier evidence with SDK-file citations.

### S3 — Contract derivation (JSON Schema → Zod)
- Port `jsonSchemaToZod` + `projectContract` from
  `integration_stitcher/packages/contracts/src/jsonschema.ts` / `projection.ts`: JSON Schema to
  Zod with `$defs` resolution, field projection to the selected surface (small tool surface,
  not raw API sprawl).
- Assemble the bubble params schema as a discriminated union on `operation` (one branch per
  OperationDraft), `credentials` field appended per BubbleLab convention; result schemas union
  the per-operation outputs with `success`/`error` envelope, matching existing bubbles.
- Every leaf carries `.describe()` filled from doc prose, because `.describe()` is what the
  codegen LLM and `get-bubble-details-tool` read downstream.

### S4 — Side-effect classification
Exactly BACKFILL-METHOD, executed inside the pipeline:
- Evidence assembled per operation in hierarchy order (mcp > openapi > prose), fed to the
  shipped `classifySideEffect` (`packages/bubble-core/src/utils/side-effect-classifier.ts`).
- No signal → fail-safe `write` + `unverified: true` + confidence 0.2, queued for human review.
- Carrier archetype: the carrier operation records fail-safe `write` with a carrier-note
  citation, plus a generated statement-level refinement hook (SQL verb allowlist) per
  BACKFILL-METHOD §7.4.
- Output is the colocated `<app>.metadata.ts`, same generator as the shipped backfill script.

### S5 — Auth inference
From OpenAPI `securitySchemes` / MCP server auth / prose auth pages, mapped onto BubbleLab's
credential system (reference logic: `integration_stitcher/packages/contractgen/src/auth-infer.ts`):

| Doc evidence | authType | CredentialType + config |
|---|---|---|
| `apiKey` in header/query; `http: bearer`; `http: basic` | `'apikey'` | `<APP>_API_KEY` (or `_TOKEN`), placement recorded in the request builder |
| `oauth2` authorizationCode flow | `'oauth'` | `<APP>_CRED`, authorization/token URLs + scope map into the oauth config |
| DB-style connect (host/account + user + secret, JWT keypair) | `'apikey'` with `credentialConfigurations` extra fields (the PostgreSQL `ignoreSSL` mechanism generalizes) or `'connection-string'` | e.g. Snowflake: account URL + PAT/JWT; Databricks: workspace host + PAT |

Rules: client secrets are never inlined (env-var names derived from the app id,
`SNOWFLAKE_CLIENT_ID` pattern); multi-scheme apps emit ONE bubble with the scheme choice
resolved in `chooseCredential()`; scopes land in `requiredScopes` per operation (feeds IR-6/7).

### S6 — Code generation
Template-driven emission of the folder in §1. The only non-boilerplate parts are:
- the per-operation `toRequest` binding table (from S2 wire bindings) feeding one shared
  `buildRequest` util (port of `generate.ts` `buildToRequest`: path substitution, query, headers,
  JSON body, Content-Type);
- `performAction` switch: validate → build request → fetch → parse → validate against result
  schema → `{success, error, …}` envelope;
- `testCredential()`: generated to call the cheapest `read`-classified operation;
- `chooseCredential()`: keyed to the credential types from S5.

No `as any` anywhere in emitted code; emitted files must pass the repo lint + typecheck as-is.

### S7 — Validation gates (ordered, all must pass)
1. **Static**: `pnpm typecheck` + `pnpm lint:check` on the generated package surface.
2. **Schema guard**: every schema operation has a metadata entry with non-empty
   source+citation (extends the shipped `operation-metadata.test.ts` coverage guard).
3. **Factory**: bubble registers, `factory.getMetadata()` exposes params/result/operationMetadata.
4. **Round-trip**: generated example params (from doc examples where present) parse through the
   params schema; recorded/spec response examples parse through result schemas.
5. **Human review gate** (mandatory for prose-derived drafts, and for any `unverified`
   classification): a diff-shaped review of operations + classifications + citations.
6. **Live probe (Phase-2 semantics, credential required)**: `testCredential()` plus at most the
   selected `read` operations run for real; `write`-classified operations are NEVER executed
   here (two-phase model, REPO-MAP §4b; the test-mode switch enforces it at the base class).
   Real read responses are recorded (the Contract KB seed, IR-11).

### S8 — Registration (the 12-location problem)
A codemod (`scripts/register-bubble.ts`) applies the CREATE_BUBBLE_README checklist mechanically;
§6 REPO-REFLECTION lists every file. Near-term the codemod is honest about the 12 locations;
the IR-1 declarative AppSpec is the long-term fix that collapses them (SURGERY, separately
scheduled). The codemod is idempotent (re-running for an existing app is a no-op diff).

## 4. Worked example A — Snowflake (official OpenAPI exists)

Facts (verified): official specs at `snowflakedb/snowflake-rest-api-specs`;
`specifications/sqlapi.yaml` is OpenAPI 3.0.0 for the SQL API v2; resource APIs
(database.yaml, warehouse.yaml, task.yaml, …) ship alongside; prose docs at
docs.snowflake.com; the Snowflake Python API is generated from these specs.

Pipeline walk:
- **S1**: spec repo hit at step 2 of discovery. DocPack = sqlapi.yaml (+ selected resource specs).
- **S2** (deterministic): from sqlapi.yaml:
  `execute_statement` (POST `/api/v2/statements`, "Submits one or more statements for
  execution"), `get_statement_status` (GET `/api/v2/statements/{statementHandle}`, "Checks the
  status of the execution"), `cancel_statement` (POST `/api/v2/statements/{statementHandle}/cancel`,
  "Cancels the execution of the statement"). Optional ops-team adds from resource specs
  (`list_databases`, `list_warehouses`, …).
- **S3**: request body schema (statement, timeout, database, schema, warehouse, role, bindings,
  parameters) flattens into the `execute_statement` branch; every field's `.describe()` comes
  from the spec's own descriptions.
- **S4**: `execute_statement` = carrier → fail-safe `write` + statement-level SQL-verb
  refinement hook; `get_statement_status` → `read`; `cancel_statement` →
  `read_with_side_effects` (mutation, no record). All cited into `snowflake.metadata.ts`
  (`sqlapi.yaml#/paths/...` pointers), source `openapi`, confidence 0.85.
- **S5**: securitySchemes verified in the spec: `KeyPair` (bearer JWT +
  `X-Snowflake-Authorization-Token-Type: KEYPAIR_JWT`), `ExternalOAuth` (bearer),
  `SnowflakeOAuth` (oauth2), `ProgrammaticAccessToken` (bearer + `PROGRAMMATIC_ACCESS_TOKEN`
  header). MVP credential: `authType 'apikey'`, `SNOWFLAKE_PAT` +
  `credentialConfigurations: { accountUrl }` (the base URL is per-account:
  `https://<account>.snowflakecomputing.com`), token-type header stamped by the request builder.
  OAuth variants are additional CredentialTypes later, not blockers.
- **S6/S7/S8**: standard. `testCredential()` = `get_statement_status` is stateful, so the
  generator picks a cheap resource read (`GET /api/v2/databases`) or submits `SELECT 1` under
  the statement-refinement `read` rule with the user's sign-off.

## 5. Worked example B — Databricks (no public OpenAPI)

Facts (verified): no official public OpenAPI document; `docs.databricks.com/api/...` returns a
2.3 KB SPA shell to plain fetches (probes including `.json` variants return HTML); official SDKs
are generated from an internal spec by the "Databricks SDK Code Generator";
`databricks-sdk-go/service/sql/impl.go` shows `ExecuteStatement` = POST `/api/2.0/sql/statements`;
`service/compute/impl.go` shows cluster events = POST `/api/2.1/clusters/events` with docstring
"Retrieves a list of events about the activity of a cluster"; auth = PAT
(`DATABRICKS_TOKEN`, bearer) with OAuth/OIDC variants per the SDK README.

Pipeline walk:
- **S1**: discovery finds no spec (step 2 misses), docs crawl hits the SPA shell (step 4
  fails loudly), SDK path engages (step 5a): clone/fetch `databricks-sdk-go`, mine
  `service/<svc>/api.go` docstrings + `service/<svc>/impl.go` verb/path pairs. DocPack = the SDK
  files + the human-readable `docs.databricks.com/api/...` URLs recorded per operation for
  citations.
- **S2** (deterministic parse of generated code): for the technical-team MVP surface:
  `execute_statement` (POST `/api/2.0/sql/statements`, "Execute a SQL statement and optionally
  await its results for a specified time"), `get_statement` (poll status/result),
  `cancel_execution` ("Requests that an executing statement be canceled"),
  plus warehouse/cluster reads (`list_warehouses`, `get_cluster`, `cluster_events`).
  Input schemas come from the SDK request structs (typed, generated from the same internal
  spec); this stays prose-tier evidence because the public artifact is the SDK, not a spec.
- **S3**: SDK struct fields → JSON Schema → Zod branches (warehouse_id, statement, wait_timeout,
  disposition, format, …).
- **S4**: `execute_statement` = carrier → fail-safe `write` + SQL-verb refinement;
  `get_statement` → `read`; `cancel_execution` → `read_with_side_effects`;
  `cluster_events` → `read` (the POST-that-reads worked example, BACKFILL-METHOD §7.1).
  Source `prose`, confidence 0.6, citations = docs URL + SDK file quote.
- **S5**: `authType 'apikey'`: `DATABRICKS_TOKEN` (bearer) +
  `credentialConfigurations: { host }` (workspace URL). OAuth M2M (machine-to-machine) is a
  later CredentialType.
- **S7**: the human review gate is MANDATORY here (prose path). Live probe:
  `testCredential()` = `list_warehouses` (read).

The pair proves the pipeline's invariant: Snowflake exercises the fully deterministic spec path,
Databricks exercises the prose/SDK path with the review gate, and both land in the identical
ServiceBubble shape with cited side-effect metadata. Every future app is one of these two walks
(or the easier MCP walk).

## 7. Human-in-the-loop points (all of them, nothing hidden)

1. **Source confirmation** (S1): "found spec X / SDK Y, using it" — one click, catches wrong-app
   docs early.
2. **Operation selection** (S2, ops teams): checkbox the endpoints that become operations;
   default = a curated top slice, never all endpoints of a 900-path spec.
3. **Prose-extraction review** (S7.5, prose/SDK path only): approve the OperationDraft diff,
   quotes shown inline.
4. **Classification review** (S7.5): every `unverified` fail-safe and every carrier hook needs
   an explicit ack; cited classes are shown but need no ack.
5. **Credential entry + testCredential** (S7.6): the user pastes the token/host; the pipeline
   never sees vendor signup flows.
6. **First-run write approval** (outside this pipeline, standing two-phase gate,
   REPO-MAP §4b): write-classified call sites need per-operation sign-off on the first real run.

Everything else is automatic. Target: spec-path apps need clicks 1, 2, 5 only.

## 8. Acceptance criteria — "the app was successfully added"

1. `pnpm typecheck` and `pnpm lint:check` pass with the generated + codemodded files.
2. `factory.registerDefaults()` registers the bubble; `getMetadata('<app>')` returns params
   schema, result schema, and `operationMetadata`.
3. The coverage guard passes: every schema operation classified, every classification cited
   (non-empty source + citation), zero silent guesses (`unverified` allowed, uncited not).
4. `get-bubble-details-tool` output for the app shows `[side-effect: …]` tags per operation;
   `bubbles.json` carries the app with `operationMetadata`.
5. A credential can be added in studio without UI errors (checklist #4 wired) and
   `testCredential()` returns true against the live service.
6. One `read`-classified operation executes live and its response validates against the
   generated result schema (drift = fail, not warn).
7. No `write`-classified operation was executed at any point during onboarding.
8. A generated flow using the bubble passes `bubble-runtime` validation and renders in the
   studio visualizer.

Criteria 1–4 are CI-checkable with zero credentials; 5–8 are the credentialed smoke, run once
per onboarding.

## 9. Implementation shape and build order

New package `packages/bubble-appgen` (pipeline stages S1–S6 as pure functions + a CLI
`pnpm add-app --name snowflake --openapi <url>` / `--docs <url>` / `--sdk <repo>`), plus
`scripts/register-bubble.ts` (S8 codemod) in bubble-core. Studio "Add app" wizard calls the same
pipeline over the API. Ordering per HANDOFF §8: IR-8 is shipped (prerequisite met); this feature
is the CG item and slots after the test-mode switch, so that criterion 7 is enforced by the base
class rather than by pipeline discipline alone. Ports from the reference build (design, not
verbatim): `contractgen/src/generate.ts` (S2/S6 request mapping), `contracts/src/jsonschema.ts` +
`projection.ts` (S3), `contractgen/src/auth-infer.ts` (S5). The prose/SDK front-end (S1 steps
4–5, S2 prose path) is new work, listed in HANDOFF §10 as the true unlock, and this document is
its design.
